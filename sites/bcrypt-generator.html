Understanding Bcrypt: The Gold Standard for Password Hashing

Bcrypt is a password hashing function designed by Niels Provos and David Mazieres in 1999, based on the Blowfish block cipher. Unlike general-purpose hash functions such as MD5 or SHA-256, bcrypt was specifically engineered to be slow and computationally expensive, which is exactly what you want when protecting passwords. The core idea behind bcrypt is that verifying a legitimate user's password should take a fraction of a second at most, but an attacker trying billions of guesses should face an enormous wall of computation that makes brute-force cracking impractical.

The fundamental difference between hashing and encryption is that hashing is a one-way operation. When you encrypt data, you can reverse the process with a decryption key. When you hash a password with bcrypt, there is no mathematical shortcut to recover the original password from the hash output. The only way to determine whether a candidate password matches is to run it through the same hashing process and compare the results. This is precisely how login systems work: they store only the hash, never the plaintext password, and compare submitted passwords by hashing them fresh each time.

One of bcrypt's most important features is its built-in salt. A salt is a random value that is mixed into the hashing process before any computation begins. Because each hash gets its own unique salt, even two users with the exact same password will end up with completely different hash outputs. This eliminates an entire category of attacks: precomputed lookup tables and rainbow tables become useless because an attacker would need to build a separate table for every possible salt value, which is computationally infeasible for a 128-bit salt space.

The cost factor, sometimes called the work factor or log-rounds parameter, is what makes bcrypt an adaptive hashing algorithm. When you set a cost factor of 10, bcrypt performs 210 (1,024) iterations of the Blowfish key expansion. Increasing the cost by 1 doubles the computation time. This means that as processors get faster over the years, administrators can simply increase the cost factor to maintain the same level of brute-force resistance without requiring users to change their passwords or migrating to a different algorithm.

Why should you choose bcrypt over MD5 or SHA-256 for password storage? General-purpose hash functions are designed to be fast. A modern GPU can compute billions of SHA-256 hashes per second, which means an attacker can test billions of candidate passwords every second. Bcrypt, by contrast, is intentionally slow and memory-hard, making GPU-based parallel attacks far less effective. While newer algorithms like Argon2 offer even stronger defenses against specialized hardware attacks, bcrypt remains the most widely supported and battle-tested password hashing function available today, with native support in virtually every major programming language and framework.

Frequently Asked Questions

What is the difference between bcrypt and SHA-256 for password hashing?

SHA-256 is a general-purpose cryptographic hash function designed for speed. It can process millions of hashes per second on a single CPU and billions per second on GPUs. This speed is an asset when verifying file integrity or building blockchain systems, but it is a critical weakness for password hashing because attackers can try enormous numbers of password guesses very quickly. Bcrypt was purpose-built for passwords. It incorporates an adjustable cost factor that controls how many iterations of the Blowfish key schedule are performed, making each hash computation deliberately slow. A bcrypt hash at cost 10 takes roughly 100 milliseconds, while a SHA-256 hash completes in microseconds. That five-order-of-magnitude difference means that a brute-force attack against bcrypt-hashed passwords requires proportionally more time and resources, giving defenders a massive advantage.

What cost factor should I use for bcrypt?

The ideal cost factor depends on your specific hardware and performance requirements. A cost factor of 10 is the most common default and is generally suitable for web applications where login should complete in under 200 milliseconds. For higher-security applications such as banking or healthcare systems, a cost factor of 12 to 13 provides stronger protection at the expense of increased login latency. The best approach is to benchmark your production server: measure how long a bcrypt hash takes at various cost factors and choose the highest value that stays within your acceptable response time budget. As a rule of thumb, if a single hash takes between 100 and 500 milliseconds, you have a good balance between usability and security. Remember that each increment doubles the computation time, so the jump from cost 12 to 13 is the same as doubling your current hashing time.

What is a salt in bcrypt and why does it matter?

A salt is a random value that is generated freshly each time a password is hashed and then stored alongside the hash output. In bcrypt, the salt is 128 bits (16 bytes) long and is embedded directly in the output string, so you never need to store it separately. The salt ensures that even if two users choose the identical password, their stored hashes will be completely different. Without a salt, an attacker who compromises a database could instantly identify all accounts sharing the same password or use precomputed tables to look up common passwords. With unique salts, an attacker must perform the full bcrypt computation individually for every hash in the database, turning a potentially quick table lookup into an extremely costly per-hash brute-force effort.

Can bcrypt protect against rainbow table attacks?

Yes. Rainbow tables are massive precomputed databases that map hash outputs back to their plaintext inputs. They work against unsalted hashes because the attacker only needs one table for a given hash function. Bcrypt defeats rainbow tables in two ways. First, the 128-bit random salt means there are roughly 3.4 times 10 to the power of 38 possible salt values. An attacker would need a separate rainbow table for each salt value, which is computationally and physically impossible to store. Second, bcrypt's slow hashing makes the precomputation itself far more expensive than it would be for fast hash functions. Even if an attacker attempted to build a rainbow table for a single salt, the time and energy cost of hashing each candidate at bcrypt's work factor would be prohibitive compared to the same effort with SHA-256 or MD5.

Is bcrypt vulnerable to timing attacks?

Bcrypt implementations should use constant-time comparison when checking whether a submitted password matches a stored hash. A timing attack works by measuring tiny differences in how long a comparison takes: if the system returns "no match" faster for completely wrong passwords than for nearly-correct ones, an attacker could theoretically narrow down the correct hash byte by byte. Well-implemented bcrypt libraries always compare every byte of the computed hash against the stored hash before returning a result, ensuring that the comparison takes the same amount of time regardless of how many bytes match. This tool uses a constant-time comparison in its verify function. If you are implementing bcrypt in production code, always use the comparison function provided by your language's bcrypt library rather than writing your own string equality check.

How does Argon2 compare to bcrypt?

Argon2 is a newer password hashing algorithm that won the Password Hashing Competition in 2015. Its main advantage over bcrypt is that it is memory-hard: in addition to requiring significant CPU time, Argon2 forces the hashing process to use a configurable amount of RAM, making attacks with specialized hardware like ASICs and FPGAs much harder. Bcrypt's Blowfish-based algorithm requires relatively little memory (about 4 KB), which means attackers with custom chips can run many parallel bcrypt computations on a single device. Argon2, configured to use 64 MB or more per hash, limits parallelism because each hashing operation must reserve that memory. Despite Argon2's theoretical advantages, bcrypt remains an excellent choice for most applications because it has decades of real-world deployment, extensive auditing, and first-class library support in virtually every language and framework. If your platform supports Argon2 natively, it is worth considering, but bcrypt is not considered outdated or insecure.

Why does bcrypt have a 72-byte password limit?

Bcrypt's 72-byte limit comes directly from the underlying Blowfish cipher. The Blowfish key schedule accepts keys up to 72 bytes (or 56 bytes in some implementations), and bcrypt feeds the password into this key schedule during the expensive key setup phase. Any password bytes beyond position 72 are silently ignored. In practice, 72 bytes accommodates passwords of 72 ASCII characters, which provides more than enough entropy for virtually any realistic password. If you need to support longer passphrases, a common workaround is to first hash the password with SHA-256 or SHA-512 to produce a fixed-length digest, then pass that digest to bcrypt. This approach preserves all of the password's entropy while fitting within bcrypt's input limit. Many modern bcrypt libraries handle this pre-hashing automatically.
